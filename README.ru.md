# logwatch

Утилита командной строки для **анализа логов и отчётов по безопасности**, без
сторонних зависимостей, написанная на чистом Python (только стандартная
библиотека). Направьте её на access-логи NGINX/Apache или auth-логи `sshd` в
Linux — и получите понятный, цветной, разбитый на секции отчёт по безопасности
и эксплуатации: топ источников, состояние по кодам ответа,
**обнаружение подозрительных запросов** (обход каталогов, SQL-инъекции, сканеры,
…) и **обнаружение перебора паролей по SSH**. Плюс машиночитаемый вывод
`--json`, готовые к публикации отчёты в **HTML / Markdown**, список доверенных
адресов (**allow-list**) и настраиваемый, удобный для автоматизации
**код возврата**.

Это единый самодостаточный пакет. Никакого `pip install`. Скопируйте его на
любую машину с Python 3.10+ и запускайте.

```text
$ python -m logwatch samples/access.log
HTTP ACCESS LOG REPORT
======================
  Lines read             42
  Parsed requests        42
  Unique client IPs      9
  Total bytes sent       5.2 MB
  Peak req/min           12 at 2024-10-10 13:55
  ...
Suspicious requests (9)
=======================
  By category:
    Scanner / recon path  8
    SQL injection pattern 3
    Path traversal (../)  2
    Unusual HTTP method   2
  ...
Risk summary
============
   RISK: MEDIUM
```

## Зачем это нужно

Когда вы заходите по SSH на сервер после инцидента, первое, что вы делаете, —
это `grep`, `awk` и `sort | uniq -c` по `access.log` и `auth.log`. Эта утилита
упаковывает такие привычные действия в одну команду, добавляет сверху
эвристики безопасности и работает одинаково везде, потому что у неё **нет
зависимостей**. Это портфолио-проект, ориентированный на работу в области
сетей / безопасности / DevOps.

## Возможности

- **Два формата логов с автоопределением:**
  - **combined access-логи** NGINX / Apache
  - **auth-логи `sshd`** в Linux (`/var/log/auth.log`)
- **Отчёт по access-логу:** общее число запросов, уникальные IP, топ
  IP / путей / user-agent, разбивка по кодам ответа (2xx/3xx/4xx/5xx),
  самые большие ответы, общий объём отданных байт и **пик запросов в минуту**.
- **Обнаружение подозрительных запросов** (эвристики на регулярных
  выражениях), каждый помечен IP-адресом источника:
  - обход каталогов (`../`, в том числе закодированные варианты)
  - полезные нагрузки в духе SQL-инъекций (`UNION SELECT`, `OR 1=1`,
    `; DROP TABLE`, …)
  - попытки межсайтового скриптинга (`<script>`, `onerror=`, …)
  - типичные пути сканеров / разведки (`/wp-admin`, `/.env`, `/phpmyadmin`,
    `/.git/`, `/cgi-bin/`, …)
  - чрезмерно длинные строки запроса
  - необычные / недопустимые HTTP-методы
- **Отчёт по auth-логу:** неудачные и успешные входы, **обнаружение перебора
  паролей** (IP, превысившие порог неудач внутри скользящего временно́го окна),
  перебираемые несуществующие пользователи, топ атакующих IP и сводка по
  успешным входам.
- **Allow-list** (`--allow` / `--allow-file`): перечислите свои заведомо
  доверенные IP и CIDR-диапазоны (внешний адрес офиса, проба мониторинга,
  раннер CI) — и их срабатывания будут подавлены и подсчитаны отдельно, чтобы
  настоящие атаки бросались в глаза.
- **Отчёты для публикации:** `--html report.html` создаёт чистый,
  самодостаточный HTML-файл (встроенный CSS, без JavaScript, с поддержкой
  тёмной темы), который можно отправить письмом или приложить к тикету;
  `--md report.md` создаёт Markdown в стиле GitHub для задач, вики или сводки
  в CI.
- **Вывод `--json`** для передачи в другие инструменты.
- **Настраиваемый код возврата по уровню риска** (`--fail-on low|medium|high`),
  чтобы работать как «ворота» в cron / CI (см. ниже).
- **Поддержка `.gz`** (определяется по сигнатурным байтам, а не только по
  расширению), **несколько файлов** и **stdin** (`-`). Повреждённые/обрезанные
  файлы и мусорные строки сопровождаются предупреждением и пропускаются —
  запуск никогда не падает.

## Требования

- Python **3.10+**
- Больше ничего. Файл `requirements.txt` намеренно пуст.

## Установка / запуск

Устанавливать нечего. Склонируйте репозиторий и запустите пакет:

```bash
python -m logwatch samples/access.log
python -m logwatch --format auth samples/auth.log
```

## Использование

```text
python -m logwatch [options] LOGFILE [LOGFILE ...]

позиционные аргументы:
  LOGFILE               один или несколько лог-файлов (поддерживается .gz);
                        используйте '-' для stdin

опции:
  --format {auto,access,auth}
                        формат лога; 'auto' определяет содержимое (по умолч.: auto)
  --top N               показать топ-N записей в каждой ранжированной секции
                        (по умолч.: 10)
  --since WHEN          игнорировать записи раньше указанного времени
                        (напр. '2024-10-10' или '2024-10-10 13:00:00')
  --threshold N         auth: число неудачных входов в окне для пометки IP как
                        перебора (по умолч.: 5)
  --window MIN          auth: скользящее окно перебора в минутах (по умолч.: 5)
  --max-query-len N     access: длина строки запроса, выше которой запрос
                        считается подозрительным (по умолч.: 512)
  --allow IP/CIDR       считать этот IP или CIDR заведомо доверенным и подавлять
                        его срабатывания; можно указывать несколько раз
  --allow-file PATH     прочитать дополнительные записи allow-list (по одному
                        IP/CIDR в строке, допускаются комментарии '#') из файла
  --json                выводить машиночитаемый JSON вместо текстового отчёта
  --html FILE           дополнительно записать самодостаточный HTML-отчёт в FILE
  --md FILE             дополнительно записать Markdown-отчёт в FILE
  --fail-on {low,medium,high}
                        минимальный уровень риска, при котором код возврата
                        становится ненулевым, для «ворот» CI (по умолч.: medium)
  --color / --no-color  принудительно включить или отключить цветной вывод
  --version             показать версию и выйти
```

### Примеры

```bash
# Автоопределение формата и печать цветного отчёта
python -m logwatch samples/access.log

# Принудительно auth-парсер, ужесточить правило перебора до 10 неудач / 2 мин
python -m logwatch --format auth --threshold 10 --window 2 samples/auth.log

# Анализ ротированного сжатого лога с выводом JSON, только топ-5
python -m logwatch --json --top 5 samples/access.log.gz

# Подавить свои IP мониторинга / офиса, чтобы настоящие атаки были заметнее
python -m logwatch --allow 10.0.0.0/8 --allow 198.51.100.7 samples/access.log

# Хранить доверенные источники в файле (по одному IP/CIDR в строке, # — коммент.)
python -m logwatch --allow-file allow.txt /var/log/auth.log

# Записать готовые к публикации отчёты HTML и Markdown рядом с выводом в терминал
python -m logwatch --html report.html --md report.md samples/access.log

# Поток из stdin (напр. живой tail или удалённый сервер)
ssh server 'cat /var/log/auth.log' | python -m logwatch --format auth -

# Завершать шаг CI с ошибкой только при риске HIGH; MEDIUM лишь сообщается
python -m logwatch --fail-on high --format auth /var/log/auth.log
```

## Коды возврата

logwatch рассчитан на использование в cron-задаче или шаге CI:

| Код  | Значение                                                                  |
|------|---------------------------------------------------------------------------|
| `0`  | Отчёт построен; риск **ниже** порога `--fail-on`.                         |
| `2`  | Отчёт построен; риск **достиг или превысил** `--fail-on` (по умолч. MEDIUM).|
| `1`  | Ошибка использования / ввода (некорректный `--since`, нет читаемых строк, …).|

`--fail-on` управляет чувствительностью «ворот»: `--fail-on low` завершается
ошибкой при любом срабатывании, `--fail-on high` — только при серьёзном. По
умолчанию — `medium`.

```bash
# Громко «уронить» конвейер, если в сегодняшних логах видна атака
python -m logwatch --format auth /var/log/auth.log || echo "review the logs!"
```

## Allow-list (доверенные источники)

Реальные логи полны *вашей собственной* инфраструктуры: внешний IP офиса,
проба доступности, раннер CI, API-клиент партнёра. Такие хосты постоянно
задевают эвристики сканеров / длинных запросов (проверки здоровья, большие
выгрузки, пинги) и топят настоящие атаки в шуме.

`--allow` и `--allow-file` позволяют перечислить эти источники один раз. Каждая
запись — это одиночный IP (`198.51.100.7`) или CIDR-диапазон (`10.0.0.0/8`),
IPv4 или IPv6:

- Доверенные источники **по-прежнему учитываются** в статистике трафика (это не
  фильтр, который их скрывает) — они лишь исключаются из обнаружений
  безопасности (подозрительные запросы, перебор, топ атакующих IP).
- Число подавленных срабатываний сообщается отдельно, чтобы было видно,
  сколько было отфильтровано.
- Нечитаемые записи пропускаются с предупреждением; отсутствующий
  `--allow-file` сопровождается предупреждением, но не является фатальным.

```bash
# allow.txt
# наш мониторинг и CI
198.51.100.7
10.0.0.0/8        # весь внутренний диапазон
2001:db8:42::/48  # IPv6-диапазон офиса
```

## Подробно об обнаружениях

### Access-логи

| Категория           | Что её вызывает                                                    |
|---------------------|--------------------------------------------------------------------|
| `path_traversal`    | `../`, `..\` или закодированные варианты в пути/запросе.            |
| `sqli`              | `UNION SELECT`, `OR 1=1`, `' OR '1'='1`, `sleep()`, `; DROP TABLE`, `information_schema`, … |
| `xss`               | `<script`, `javascript:`, `onerror=`, `onload=` в URL.             |
| `scanner_path`      | Запросы к известным путям разведки (`/wp-admin`, `/.env`, `/phpmyadmin`, `/.git/`, `/cgi-bin/`, `/actuator`, …). |
| `oversized_query`   | Строка запроса длиннее `--max-query-len` (по умолч. 512 символов). |
| `bad_method`        | HTTP-метод вне обычного набора `GET/POST/HEAD/PUT/DELETE/PATCH/OPTIONS`. |

Пути проверяются и в исходном виде, **и** после однократного URL-декодирования,
поэтому полезные нагрузки, спрятанные за кодировкой вида `%2e%2e%2f`, тоже
ловятся. Один запрос может попасть сразу в несколько категорий.

### Auth-логи

**Обнаружение перебора** группирует события неудачных/несуществующих входов по
IP-адресу источника, сортирует их по времени и проводит по ним скользящее окно
в `--window` минут. Если число неудач внутри окна достигает `--threshold`
(или этого достигает суммарное число неудач от IP), IP попадает в отчёт вместе
с перечнем перебиравшихся имён пользователей. Успешные входы никогда не
считаются атаками. Строки без пригодной метки времени всё равно учитываются в
сумме, поэтому отсутствие часов никогда не скроет атакующего.

### Сводка по риску

В конце печатается намеренно простая и объяснимая метка (`LOW` / `MEDIUM` /
`HIGH`), которая управляет кодом возврата (через `--fail-on`). Она основана на
объёме и разбросе срабатываний — это не «чёрный ящик», так что её легко
интерпретировать.

## Без GeoIP / ASN-атрибуции (намеренно)

logwatch поставляется **без базы GeoIP/ASN и не делает сетевых запросов** —
именно это сохраняет нулевые зависимости, автономность и скорость. Как
следствие, утилита **не** сообщает, какой стране, сети или организации
принадлежит IP. Каждый отчёт повторяет это уведомление рядом с таблицами IP.
Чтобы атрибутировать адрес, выполните поиск самостоятельно, например:

```bash
whois 185.220.101.4          # RDAP/whois: владелец, ASN, abuse-контакт
```

…или передайте JSON-вывод в собственное обогащение данными threat-intel.

## Структура проекта

```
logwatch/
├── logwatch/                 # запускаемый пакет
│   ├── __init__.py
│   ├── __main__.py           # включает `python -m logwatch`
│   ├── cli.py                # argparse, чтение file/gzip/stdin, диспетчеризация, экспорт
│   ├── parsers.py            # парсеры строк access-лога и sshd (чистые функции)
│   ├── detectors.py          # подозрительные запросы + перебор + логика allow-list
│   ├── analyzers.py          # потоковые агрегаторы -> словари-сводки
│   ├── report.py             # цветной текстовый отчёт для человека
│   ├── exporters.py          # сборщики самодостаточных HTML- и Markdown-отчётов
│   └── colors.py             # маленький ANSI-помощник (учёт TTY/NO_COLOR)
├── samples/                  # синтетические логи, работающие «из коробки»
│   ├── access.log            # обычный трафик + строки обхода/SQLi/сканеров
│   ├── access.log.gz         # то же, сжатое gzip (демонстрирует .gz)
│   └── auth.log              # обычные входы + всплеск перебора по SSH
├── tests/                    # набор тестов на stdlib unittest (без сети / тяжёлого I/O)
│   ├── test_parsers.py
│   ├── test_detectors.py
│   ├── test_analyzers.py
│   ├── test_allowlist.py     # allow-list + risk-rank + интеграция с анализаторами
│   ├── test_exporters.py     # экспорт HTML/Markdown (корректность, экранирование)
│   └── test_cli.py
├── README.md                 # английская версия
├── README.ru.md              # этот файл (русский перевод)
├── requirements.txt          # намеренно пуст (ноль зависимостей)
└── .gitignore
```

Архитектура держит слои чётко разделёнными: **parsers** говорят, *что
произошло*, **detectors** — *подозрительно ли это*, **analyzers** агрегируют, а
**report** / **exporters** рендерят. Любой слой, кроме `cli`, чист (без I/O) —
именно это делает набор тестов быстрым и детерминированным.

## Запуск тестов

```bash
# Из корня проекта
python -m unittest discover -s tests -v

# Проверка компиляции каждого исходного файла
python -m py_compile logwatch/*.py tests/*.py
```

Набор (92 теста) покрывает парсеры, каждую категорию детекторов, логику окна
перебора, allow-list (IP/CIDR/IPv6, подавление в обоих анализаторах),
агрегаторы, автоопределение формата, экспортёры HTML/Markdown (корректность и
HTML-экранирование враждебных полей) и CLI целиком (включая ввод gzip,
восстановление при повреждённом gzip, вывод JSON, флаги allow-list и код
возврата `--fail-on`).

## Этика и область применения

**Анализируйте только те логи, которыми вы владеете или на проверку которых
явно уполномочены.** Эта утилита — для защиты ваших собственных систем: чтения
ваших access/auth-логов, чтобы замечать атаки и эксплуатационные проблемы.
Синтетические логи в `samples/` используют документационные/зарезервированные
диапазоны IP (`203.0.113.0/24`, `198.51.100.0/24`, `192.0.2.0/24`) и не
содержат реальных персональных данных. Не направляйте logwatch на
инфраструктуру, которой вы не управляете, и относитесь к любым IP/именам
пользователей в реальных логах как к чувствительным данным: никогда не
вставляйте боевые логи в общие задачи или чаты.
